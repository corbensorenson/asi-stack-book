# Proof Adequacy Review

Last updated: 2026-07-01

This review classifies the 125 implemented Lean proof targets by adequacy for the claims they are attached to. It does not change any support state and does not claim broad system correctness.

## Summary

| Adequacy class | Targets | Meaning |
|---|---:|---|
| adequate finite-record invariant | 9 | The current Lean predicate is adequate for a narrow repository/process or finite-record claim if the claim text remains narrow. |
| useful but too narrow | 49 | The predicate is a useful guard, but the chapter needs state-machine, trace, integration, empirical, or benchmark tests before the proof can support the larger boundary. |
| needs richer state-machine or review semantics | 19 | The current finite record omits timing, lifecycle, review, social, adversarial, or governance semantics that are central to the chapter. |
| needs executable tests first | 31 | Behavior, replay, routing, context, tool-use, memory, or artifact mechanics need deterministic harnesses before stronger proof work is meaningful. |
| needs empirical or baseline tests first | 11 | Performance, efficiency, policy, substrate, or quality claims need workloads, baselines, measurements, and negative controls before proof adequacy can rise; deterministic accounting harnesses may check record discipline but do not replace model or workload results. |
| research-agenda until artifact import | 6 | The proof target remains a placeholder guard until external/local project artifacts are imported, built, replayed, or otherwise inspected. |

All 125 targets still build as narrow finite-record predicates. The review changes the project backlog, not the theorem status.

## Review Rules

- Lean build success means the checked finite predicates compile.
- A finite predicate is not automatically an adequate formalization of the chapter boundary.
- Proof adequacy cannot promote a claim unless the claim is scoped to what the proof actually checks.
- Schemas validate record shape, tests validate behavior, benchmarks validate measured claims, and source notes support drafting context. These lanes remain separate.
- `proofs/proof_triage.json` records routing into Lean, not adequacy of the resulting formalization; update it only when a proof target is added, retired, or rerouted.

## Proof Depth Classifier

`docs/proof_depth_classification.md` is generated by
`python3 scripts/validate_proof_depth.py --write` and checked by
`python3 scripts/validate_proof_depth.py`. It classifies Lean theorem bodies by
proof shape, not semantic adequacy. The current baseline is 448 theorem
declarations: 121 direct/projection-style, 327 derived/decomposed, and 0
unknown or mixed. The v1.0 safety-critical modules now contain 69 theorem
declarations: 10 remain direct/projection-style; all five safety-critical
modules have at least one derived/decomposed finite-record theorem,
`AsiStackProofs.Alignment` now has four,
`AsiStackProofs.Corrigibility` now has sixteen, and
`AsiStackProofs.GovernanceRights` now has sixteen,
`AsiStackProofs.ValueConflict` now has four, while
`AsiStackProofs.SelfImprovement` now has nineteen. Outside the safety-critical
set, `AsiStackProofs.ClaimLedger` now has a third target and fifteen
derived/decomposed theorem declarations after the revision-lifecycle route
increment. `AsiStackProofs.EvidenceStates` now has a third target and eighteen
derived/decomposed theorem declarations after the transition-lifecycle route
increment.

This does not invalidate the Lean build or the proof target manifest. It
records that the safety-critical modules are traceability hooks until they are
either upgraded with richer state, transition, lifecycle, or review semantics,
or explicitly scoped in the relevant chapter limitation prose as
projection-only finite-record claims. No support state changes were made.

## Follow-Through Increments

### Stack Boundary Trace Envelope

The opening-stack follow-through increment strengthened
`AsiStackProofs.StackBoundaries` without changing proof targets or support
states. The module now includes a finite `StackTraceStep` record,
`StackTraceValid` predicate, and a derived negative-case theorem showing that a
valid modeled trace cannot contain an external-action step when the layer lacks
external-action authority and the handoff itself is unauthorized.

This keeps `asi-is-a-stack-not-a-model` in the `useful but too narrow` class.
The theorem is a better stack-level guard than a single-field projection, but
it still does not prove whole-system safety, deployed layer enforcement,
source-to-layer traceability, integration behavior, or that the ASI Stack is
empirically better than monolithic systems.

### Authority Decision Envelope

The first follow-through increment strengthened `AsiStackProofs.Authority` beyond the initial ceiling and missing-grant predicates. The module now includes a finite `AuthorityDecisionRecord` and `AuthorityDecisionValid` predicate for modeled allow, deny, and escalate records. The new theorems check that valid modeled decisions retain audit and non-claim fields; allowed decisions carry effect receipts and do not exceed caller or active ceilings; denied decisions carry no effect receipt; and escalation routes to review. The predicate also requires allowed decisions to reject expired or revoked grants.

This keeps the `system-boundaries-and-authority` proof cluster in the `useful but too narrow` class. The predicate is closer to the synthetic authority harness, but it still does not prove runtime adapter enforcement, deployed permission checks, revocation propagation, confused-deputy resistance in a live tool wrapper, or source interpretation adequacy.

### Failure Incident Route Envelope

The Failure Modes follow-through increment strengthened
`AsiStackProofs.FailureModes` without changing proof targets or support states.
The module now includes a finite `FailureIncidentRoute`,
`FailureIncidentRecord`, and `FailureIncidentRouteFor` decision function over
authority overreach, tainted context, context authority grants,
subject-modified evaluators, requested claim promotion, and verification
status. The new theorems check that authority over a modeled ceiling routes to
authority review, tainted context without an authority grant routes to
quarantine, a subject-modified evaluator routes to frozen review, and requested
claim promotion with failed verification routes to blocked promotion.

This moves `failure-modes-of-ungoverned-intelligence` from `needs richer
state-machine or review semantics` to `useful but too narrow` because the
chapter now has protocol fixture coverage plus a narrow incident-route envelope
for the four planned scenario rows. It still does not implement a runtime
authority gate, context-admission detector, evaluator-independence probe,
claim-verification harness, failure-receipt suite, recurrence-escalation trace,
deployed failure radar, or evidence that the stack prevents the named failure
modes.
