# Proof Adequacy Review

Last updated: 2026-07-15

This review classifies implemented Lean proof targets by adequacy for the claims they are attached to. It does not change any support state and does not claim broad system correctness.

## Summary

| Adequacy class | Targets | Meaning |
|---|---:|---|
| adequate finite-record invariant | 21 | The current Lean predicate is adequate for a narrow repository/process or finite-record claim if the claim text remains narrow. |
| useful but too narrow | 203 | The predicate is a useful guard, but the chapter needs state-machine, trace, integration, empirical, or benchmark tests before the proof can support the larger boundary. |
| needs richer state-machine or review semantics | 19 | The current finite record omits timing, lifecycle, review, social, adversarial, or governance semantics that are central to the chapter. |
| needs executable tests first | 35 | Behavior, replay, routing, context, tool-use, memory, or artifact mechanics need deterministic harnesses before stronger proof work is meaningful. |
| needs empirical or baseline tests first | 18 | Performance, efficiency, policy, substrate, or quality claims need workloads, baselines, measurements, and negative controls before proof adequacy can rise; deterministic accounting harnesses may check record discipline but do not replace model or workload results. |
| research-agenda until artifact import | 2 | The proof target remains a placeholder guard until external/local project artifacts are imported, built, replayed, or otherwise inspected. |

The reviewed targets still build as narrow finite-record predicates. The review changes the project backlog, not the theorem status.

## Review Rules

- Lean build success means the checked finite predicates compile.
- A finite predicate is not automatically an adequate formalization of the chapter boundary.
- Proof adequacy cannot promote a claim unless the claim is scoped to what the proof actually checks.
- Schemas validate record shape, tests validate behavior, benchmarks validate measured claims, and source notes support drafting context. These lanes remain separate.
- `proofs/proof_triage.json` records routing into Lean, not adequacy of the resulting formalization; update it only when a proof target is added, retired, or rerouted.

## Proof Depth Classifier

`docs/proof_depth_classification.md` is generated by
`python3 scripts/validate_proof_depth.py --write` and checked by
`python3 scripts/validate_proof_depth.py`. It classifies Lean theorem bodies by
proof shape, not semantic adequacy. Current proof-depth snapshot: 298 proof targets, 91 Lean modules, 1253 theorem declarations, 885 derived/decomposed, 214 direct/projection, 154 unknown/mixed, and 1/1 safety-critical chapter classifications present.

This does not invalidate the Lean build or the proof target manifest. It
records that the safety-critical modules are traceability hooks until they are
either upgraded with richer state, transition, lifecycle, or review semantics,
or explicitly scoped in the relevant chapter limitation prose as
projection-only finite-record claims. No support state changes were made.
