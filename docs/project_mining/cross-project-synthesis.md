# Historical AI Projects: Cross-Project ASI Stack Synthesis

Date: 2026-07-10

Projects included:

- Compiled Cognitive Architecture (`cca`)
- MoECOT Manifest (`moecot-manifest`)
- BeastBrain
- BugBrain
- Corben's Trainer
- Corben's Best Model Possible

## Executive decision

The six mined projects contain a coherent architecture lineage, not six
independent confirmations. BeastBrain is the earliest broad implementation
lineage for durable knowledge lattices, semantic coordinates, planning,
organism-style background cognition, SSD-first memory, handles, tribunals, and
personal compute. CCA turns several of those concepts into a compiler-governed
cognitive stack with append-only epistemic revision, trace bundles, protected
self-modification, formal coupling, and benchmark-truth inheritance. MoECOT
extends the compiler/control-plane pattern across registries, targets,
delegation, context, memory, routing, multi-generation training, maintenance,
and bounded autonomous growth. BugBrain compresses and operationalizes parts of
the lineage into hardware-explicit ownership, compact ontologies, signed tiered
context, durable privileged-action state, protocol security, and concrete
readiness machinery. Corben's Trainer supplies the experiment and evidence
institution: typed plans, campaigns, artifact lineage, learning-truth gates,
benchmark modes, claim evaluation, quarantine, and promotion. Corben's Best
Model Possible supplies the recurrent/adaptor/search laboratory and the
sharpest implementation audit of whether named subsystems causally affect
runtime behavior.

The central idea for the ASI Stack is therefore not any one project name. It is
a governed chain from intent and authority through compilation, context,
model/tool execution, verification, artifacts, evaluation, belief change, and
promotion. Every link must carry both an interface contract and evidence that
the declared effect occurred.

The projects also converge on a common failure pattern:

> Architecture-shaped structure is easy to produce. Capability truth requires
> causal linkage, material effects, independent evaluation, and transitive
> revocation.

This synthesis recommends strengthening existing chapters with a shared set of
cross-cutting laws. It also confirms a genuine missing boundary—durable
semantic memory and knowledge lattices—but keeps the manifest change deferred
until independent literature is ingested and the boundary is checked against
the Virtual Context ABI, procedural memory, claim ledgers, and context
transactions.

No project result is promoted to empirical support. The six repositories are
implementation references, historical design lineage, and negative evidence.

## Provenance rules for using the projects

### One lineage is one line of support

Repeated terminology and mechanisms across BeastBrain, CCA, MoECOT, BugBrain,
Trainer, and Best Model usually represent inheritance, porting, or refinement.
They do not count as independent replication. A chapter may cite multiple
projects to explain evolution, but its support ledger should treat the family
as one local implementation lineage unless independently generated evidence is
established.

### Source roles

This synthesis uses five provenance roles:

- **primary lineage:** earliest coherent implementation among the inspected
  projects;
- **refinement:** later semantic, governance, or interface development;
- **operationalization:** translation into concrete resource, hardware, or
  runtime constraints;
- **control plane:** governance over experiments, artifacts, evidence, or
  promotion;
- **negative case:** implementation behavior that narrows a claim or reveals a
  missing invariant.

A project can occupy more than one role. None of these roles means independent
external validation.

## The cross-project laws to add to the ASI Stack

### 1. The name-to-effect law

A type, module, interface, report field, or architecture diagram does not prove
that the named mechanism affects a decision or output. Every important
subsystem needs an effect trace:

`request -> derived input -> policy/state selection -> executed effect -> receipt -> evaluator`

This law is supported negatively by BeastBrain's defined-but-bypassed modules,
BugBrain's decision-invariant epistemic terms, Best Model's default/empty
subsystem integration, and Trainer's truth-shaped fields whose upstream facts
are missing.

Book actions:

- add an effect-map section to `integrated-reference-architecture`;
- require ablations for every named objective term in
  `benchmark-ratchets-and-anti-goodhart-evidence`;
- require runtime-binding receipts in
  `evidence-states-and-claim-discipline`.

### 2. The canonical-truth separation law

Source manifests, canonical registries, durable run facts, runtime caches,
operator views, narrative reports, and public claims are different projections.
Views may be regenerated; canonical facts must not be overwritten by a more
attractive summary.

BeastBrain and BugBrain supply narrative/artifact contradictions. MoECOT
explicitly distinguishes canonical graph truth from disposable read models.
Trainer derives reports from run facts but retains stale claim/promotion state
after quarantine. Best Model's promoted bank fails when its stored location is
mistaken for durable identity.

Book actions:

- define canonical state and projection classes in
  `artifact-graphs-audit-logs-and-replay`;
- require machine-readable artifacts to outrank same-revision narrative in
  `evidence-states-and-claim-discipline`;
- add transitive refresh/revocation duties to
  `artifact-steward-agents-and-living-project-governance`.

### 3. The causal capability-chain law

Optimization evidence becomes capability evidence only through a same-lineage
chain:

`data + objective + code + environment -> updated state -> selected checkpoint -> runtime decode/action -> raw output -> independent evaluator -> claim`

CCA shows falling loss without benchmark capability. BeastBrain shows simulated
curves and a router loss without parameter updates. BugBrain shows cognitive
interfaces without trained effects. Trainer introduces learning-truth gates but
can still trust producer evidence. Best Model shows a trained adapter whose
logits are not established as the source of returned text.

Book actions:

- add a trainable-state manifest and response-causality receipt to
  `evidence-states-and-claim-discipline`;
- add actual mutation, fixed-probe effect, forgetting, and rollback to
  `data-engines-continual-learning-and-unlearning`;
- add checkpoint-to-output binding to
  `benchmark-ratchets-and-anti-goodhart-evidence`.

### 4. The evidence-cell law

Evidence should be classified on at least four independent axes:

1. **origin:** measured, derived, recomputed, modeled, fixture, constant,
   declared, proxy, vacuous, or unknown;
2. **method:** syntax, deterministic recomputation, property test,
   metamorphic test, differential implementation, independent solver/prover,
   empirical measurement, or human review;
3. **independence:** shared or independent data, code, artifacts, algorithm,
   model, operator, and funding/authority source;
4. **effect:** interface present, exercised, material state changed, durable
   across restart, end-to-end outcome changed, or externally reproduced.

The result is an evidence cell, not a single green/red bit. Best Model provides
the clearest metric-origin failure; CCA and MoECOT provide formal-depth and
holdout boundaries; BeastBrain and BugBrain provide simulation/theory-label
failures; Trainer provides producer-declaration and authenticity failures.

Book actions:

- make the evidence cell a core construct in
  `evidence-states-and-claim-discipline`;
- bind admissible evidence cells to claims in
  `claim-evidence-matrix`;
- require missing origin to resolve to unknown, never to real, in
  `benchmark-ratchets-and-anti-goodhart-evidence`.

### 5. The material-transition law

A named transition is not complete until its postcondition is observable in
canonical state and survives the relevant failure/restart boundary. This
applies to memory freeze/thaw, belief revision, quarantine, promotion,
checkpoint completion, context registration, tier rebalance, compaction,
approval, revocation, and rollback.

Book actions:

- require postcondition receipts and restart tests in
  `context-transactions-snapshots-mounts-and-taint`;
- require acknowledged async terminal results in
  `artifact-graphs-audit-logs-and-replay`;
- require quarantine and revocation closure in
  `readiness-gates-residual-escrow-and-quarantine`.

### 6. The unified-authority-tuple law

Approval, quota, policy, target state, execution, artifact, trace, replay, and
revocation must bind the same authority identity. A cache key containing only a
run ID, command class, or scenario acknowledgement cannot safely authorize a
privileged action.

The minimum tuple is:

`principal + role + run + action + target + target-state digest + parameters + policy version + budget + expiry + nonce`

Book actions:

- add the tuple to `system-boundaries-and-authority`;
- use it in `intent-to-execution-contracts` and
  `runtime-adapters-tool-permissions-and-human-approval`;
- connect it to hardware and secret custody without claiming that protocol
  cryptography supplies a hardware root of trust.

### 7. The transitive-revocation law

When an upstream source, benchmark, checkpoint, evaluator, or run is
quarantined, every dependent evidence bundle, claim, promotion pointer,
readiness state, report, and public surface must become stale or revoked unless
an alternate valid derivation is proven.

CCA's closure inheritance and Trainer's stale post-quarantine claims make this
an immediate chapter addition. MoECOT's canonical registries and BugBrain's
artifact identity make the dependency graph operational.

Book actions:

- add reverse dependency closure to `artifact-graphs-audit-logs-and-replay`;
- add automatic claim/promotion invalidation to
  `benchmark-ratchets-and-anti-goodhart-evidence`;
- require revocation leases and expiry in readiness/deployment chapters.

### 8. The semantic-capacity ladder

Capacity claims must distinguish:

- addressable;
- formatted;
- resident;
- active;
- populated;
- trainable;
- updated;
- reachable by runtime;
- useful under a specified task/evaluator.

This ladder applies to memory nodes, context length, model parameters, adapter
states, storage pages, expert banks, corpus records, and hardware resources.
BugBrain supplies the physical-capacity counterexample; BeastBrain the
large-memory claims; Trainer the requested/resolved/observed resource split;
Best Model the fixed/trainable/updated state split.

Book action: add the ladder to `resource-economics-and-token-budgets` and reuse
it in model, memory, and benchmark chapters.

### 9. The router evidence ladder

Routing evidence progresses through:

1. routes/specialists exist;
2. route features are request-derived and provenance-bearing;
3. oracle labels select the intended route;
4. a learned or rule policy selects on held-out requests;
5. fallback and abstention are calibrated;
6. budget persists across the intended owner/lifetime;
7. route selection improves end-to-end outcome against fixed-compute
   baselines;
8. receipts prove execution and interference is measured.

BeastBrain stops before real parameter update; CCA/MoECOT provide typed policy
and health concepts; Best Model reaches explicit specialist banks and oracle
binding but not autonomous selection; Trainer can govern the experiment.

Book action: add this ladder to `routing-heads-and-specialist-cores`.

### 10. The negative-knowledge frontier law

An improvement engine should preserve why a candidate failed, not only which
candidate won. Retired families, failed axes, invalid metrics, duplicated
attempts, contamination, harness failures, resource exhaustion, and
best-versus-final drift are reusable knowledge.

Book actions:

- add a configuration frontier with parent, mutation, matched budget,
  protected metrics, result, rejection reason, retirement scope, and
  recombination policy to `open-ended-improvement-engines`;
- add retry ceilings and “new intervention required” rules to
  `benchmark-ratchets-and-anti-goodhart-evidence`;
- add architecture-debt and evidence-debt budgets to
  `artifact-steward-agents-and-living-project-governance`.

### 11. The formal-semantic-depth law

Formal checks should record not only theorem and build status but also
abstraction map, trusted assumptions, finite-domain scope, properties true by
construction, excluded runtime behavior, and the exact implementation surface
bound to the result. Digest equality, source mapping, or theorem count is not
semantic preservation.

Book actions:

- add semantic-depth audits to
  `executable-specifications-and-lean-proof-envelope`;
- add field-lineage and counterexample tests to
  `cognitive-compilation-and-semantic-ir`;
- add verification-method and independence labels to
  `spinoza-verification-and-proof-carrying-claims`.

### 12. The honest-missingness law

Absent evidence must become `unknown`, `not_attempted`, or `not_applicable`.
It must never default to real execution, passed, deterministic, independent,
or hardware-observed. Skipped checks cannot make readiness green without an
explicit policy-backed waiver.

Book actions:

- apply this law in evidence, benchmark, readiness, resource, and deployment
  schemas;
- distinguish `not_applicable`, `not_required`, `not_attempted`, `missing`,
  `failed`, `waived`, and `passed`.

## Durable semantic memory chapter boundary

### Why the boundary is real

The live book has strong chapters for the Virtual Context ABI, context
transactions, procedural memory, claim ledgers, artifact graphs, and memory-
adjacent recurrence. None owns the durable semantic knowledge substrate itself:

- stable semantic object identity;
- typed relation ontology and migration;
- provenance-preserving merge and supersession;
- graph/vector hybrid retrieval and navigation;
- unresolved contradiction and failure retention;
- forgetting, retraction, rollback, and compaction;
- hot/warm/cold persistence and restart consistency;
- certified bounded subgraph snapshots for context materialization.

BeastBrain supplies the primary inspected DKL/Portia lineage. CCA sharpens the
durable-memory versus context boundary and adds append-only epistemic
lifecycles. MoECOT adds governance and transition contracts. BugBrain supplies
hardware-bounded graph/context operationalization. Best Model provides a
negative implementation audit of trace-only transitions and non-durable tier
changes. Trainer supplies the artifact/evidence governance needed to evaluate
memory experiments.

### Why the manifest action remains deferred

The local lineage is not enough to establish general design claims. Before
adding `durable-semantic-memory-and-knowledge-lattices` to
`book_structure.json`, the book should ingest independent primary literature
for knowledge graphs, graph-RAG, learned graph navigation, agent long-term
memory, temporal/event-sourced knowledge, vector/graph hybrid retrieval,
memory consolidation, and transactional persistence. The new chapter must
then be tested for duplication against:

- `virtual-context-abi`;
- `context-transactions-snapshots-mounts-and-taint`;
- `procedural-memory-and-cognitive-loop-closure`;
- `claim-ledgers-and-belief-revision`;
- `artifact-graphs-audit-logs-and-replay`.

Recommended location: Part II, after the Virtual Context ABI and before context
transactions, so durable state is defined before its bounded materialization
and transaction semantics.

## Cross-project contradiction ledger

### Architecture breadth versus capability

- BeastBrain and MoECOT contain very broad architectures while acknowledging
  weak external or implemented capability.
- CCA records large optimization changes without benchmark closure.
- Best Model contains model-shaped recurrence but a frozen deterministic trunk
  and no general model-driven response path.

Resolution: architecture completeness and capability evidence are different
axes. Neither may compensate for the other.

### Internal fixtures versus external validity

- BeastBrain contains simulated benchmarks and self-consistent internal tests.
- MoECOT's internal results diverge from external holdouts and speech WER.
- Trainer's external-looking benchmark runs were later quarantined as
  placeholders.
- Best Model's speech round trip uses a private tone codec.

Resolution: fixture class, source authenticity, evaluator independence, and
external validity must be explicit claim preconditions.

### Determinism versus correctness

- CCA/MoECOT emphasize deterministic compilation and traces.
- BugBrain provides deterministic artifact replay.
- Best Model's compiler, replay, and speech paths can be deterministic while
  missing the intended semantics.

Resolution: determinism is a reproduction property, not a semantic correctness
or capability property.

### Cryptographic vocabulary versus root authority

- BeastBrain proposes handles and signed transports.
- BugBrain implements protocol security but derives some keys from non-secret
  material and does not establish a hardware root.
- Trainer uses hashes/content addressing but several journals and identities
  are non-cryptographic or incomplete.

Resolution: transport integrity, content identity, secret custody, device
identity, secure boot, rotation, anti-rollback, and hardware root are separate
claims.

### Green reports versus canonical failures

- BeastBrain readiness prose conflicts with build history.
- BugBrain narrative green state conflicts with red machine artifacts and
  skipped checks.
- Trainer retains promoted claims and a candidate pointer after quarantine.
- Best Model notes historical green checks that cannot currently be reproduced.

Resolution: canonical machine facts, freshness, requiredness, and dependency
closure govern readiness; summaries are projections.

## Minimum integrated architecture suggested by the lineage

The smallest serious implementation that captures the mined ideas is:

1. **Intent contract.** Role, objective, constraints, acceptance tests,
   non-goals, uncertainty, effects, authority, budget, and failure behavior.
2. **Compiler.** Source intent, semantic IR, executable TaskIR, pass receipts,
   ambiguity ledger, contract lineage, repair scope, and fail-closed errors.
3. **Canonical registries.** Versioned capabilities, tools, models, datasets,
   evaluators, hardware profiles, policies, and schemas.
4. **Durable semantic memory.** Stable objects, typed relations, provenance,
   justifications, contradiction state, supersession, compaction, and restart
   consistency.
5. **Context ABI.** Certified bounded snapshot with authority, provenance,
   generation, expiry, revocation, and taint.
6. **Router.** Request-derived features, persistent budget, selected/fallback
   path, abstention, expected gain/cost, and outcome calibration.
7. **Execution boundary.** Model, tool, hardware, file, network, code, and human
   authority enforced below the adapter API.
8. **Response/action receipt.** Actual checkpoint, tokenization, decode, tool
   calls, substitutions, side effects, and final output.
9. **Verification tribunal.** Method labels, independence graph, dissent,
   abstention, falsification, residuals, and claim scope.
10. **Artifact/evidence graph.** Content identity, role, lineage, replay grade,
    metric origins, claim links, quarantine, revocation, and relocation.
11. **Experiment institution.** Typed plans, real seed/source/data/environment
    identity, durable campaigns, learning-truth gates, checkpoints, fixed
    budgets, holdouts, and evaluator separation.
12. **Improvement frontier.** Champion/challenger, protected mutable partition,
    single-axis changes, matched budgets, negative results, debt caps, rollback,
    and stop rules.
13. **Readiness and release.** Applicability, requiredness, attempt, result,
    waiver, evidence, expiry, residual, and transitive dependency closure.

No single inspected project implements this complete architecture with
end-to-end validated effects. The value of the synthesis is that each project
defines or stress-tests a different boundary.

## Validation and evidence posture

### Reproduced during the mining program

- selected project identities, file/size/count inventories, and artifact-state
  observations;
- BugBrain kernel host tests: 227 passed, 3 ignored;
- BugBrain Python trainer tests: 79 passed;
- project-specific schema and source-note validation for the ASI Stack records;
- source-level behavior audits and selected retained-artifact calculations.

### Not reproduced

- CCA, MoECOT, BeastBrain, Trainer, and Best Model full workspaces under their
  intended dependency/toolchain environments;
- model training, MLX/GPU execution, external specialists, Pi/HIL deployment,
  federation, or sovereign mesh operation;
- external benchmark scores, speech quality, memory quality, routing utility,
  proof/runtime equivalence, or safety claims;
- complete artifact replay after earlier cleanup.

### Independence posture

The six projects share authorship, concepts, code lineage, terminology, and in
some cases artifacts or trainer/backend boundaries. They should not be counted
as independent empirical replications. Independent literature and evaluation
remain required for generalized book claims.
