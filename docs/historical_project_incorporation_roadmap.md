# Historical Project Incorporation Roadmap

Last updated: 2026-07-10

Status: active chapter-improvement roadmap; planning and source-routing only;
no support-state promotion

## Outcome

The six mined historical projects should be folded into **The ASI Stack** as a
single evolving implementation lineage, not as six promotional case studies and
not as six independent confirmations. The preferred action is to deepen the
chapters that already own the relevant interface, invariant, artifact, or
failure mode. A new chapter is permitted only when the current spine leaves a
real layer unowned and independent literature supports that boundary.

The program has four content priorities:

1. strengthen the evidence spine before importing architecture claims;
2. connect every named subsystem to a material runtime effect;
3. turn failed experiments and contradictions into reusable invariants;
4. add a durable-semantic-memory chapter only if its ownership test clears.

The source base is already mined and recorded in:

- `docs/project_mining/cross-project-synthesis.md`;
- `docs/project_mining/cca.md`;
- `docs/project_mining/moecot-manifest.md`;
- `docs/project_mining/beastbrain.md`;
- `docs/project_mining/bugbrain.md`;
- `docs/project_mining/corbens-trainer.md`;
- `docs/project_mining/corbens-best-model-possible.md`;
- the six corresponding source notes and project-mining backlog records.

## Governing decisions

### Existing chapters first

The mined mechanisms already have strong owners across the evidence,
architecture, compiler, authority, memory, routing, resource, readiness, proof,
and improvement chapters. The first three execution waves therefore update
existing chapters only. The work should add mechanisms, negative cases,
interfaces, invariants, proof/test targets, figures, and source mappings—not
additional repository-shaped chapters.

### One local lineage, not independent replication

BeastBrain is the early broad architecture; CCA and MoECOT progressively compile
that architecture into governed contracts; BugBrain operationalizes selected
boundaries under hardware constraints; Corben's Trainer governs experiments and
evidence; Corben's Best Model Possible is the model/adaptor/search laboratory
and the strongest interface-versus-effect audit. Repetition across them records
lineage and refinement. It does not raise a support state.

### Negative evidence is a first-class contribution

The projects are most valuable where retained behavior narrows an attractive
claim: simulated training, non-updating routers, proxy benchmarks, stale
promotions, report/prose contradictions, trace-only memory transitions,
adapter/trunk conflation, non-relocatable artifacts, metadata-supplied routing,
and self-consistent tone-codec speech. These should appear as concise technical
failure cases, not embarrassing footnotes or hidden caveats.

### Chapter count is not a target

The active manifest contains 54 chapters as of this update. Historical 44-,
45-, and 53-chapter states remain versioned snapshots. This roadmap neither
targets a larger table of contents nor reopens completed consolidation work.
Every active-surface count must remain derived from `book_structure.json`.

### Support states stay fixed

Project source notes support design reasoning, implementation reference, and
negative-case analysis. They do not establish external model capability,
benchmark performance, safety, secure deployment, natural speech, durable
memory quality, or recursive self-improvement. Chapter core claims remain at
their current support states unless a separate accepted evidence transition
justifies movement.

## Execution order

The order prevents the book from importing architecture-shaped claims before it
has the vocabulary to classify them honestly.

### Wave 0 — Bind the sources and vocabulary

Purpose: make the mining corpus usable by chapter work without copying private
project material or counting lineage as replication.

Required actions:

1. Add the six project source IDs only to chapters that own a specific mined
   mechanism or failure case.
2. Add a short lineage note to the book-level architecture discussion: the six
   projects are evolutionary local implementation references.
3. Normalize recurring terms before prose work:
   - durable semantic memory versus context materialization versus procedural
     memory;
   - model trunk versus generated/frozen features versus trainable adaptor;
   - specialist bank versus oracle binding versus autonomous router;
   - playback versus deterministic re-execution versus semantic replay;
   - declaration versus observed receipt;
   - protocol security versus secret custody versus hardware root of trust.
4. Keep raw paths, corpora, model/optimizer states, private logs, credentials,
   and unpublished benchmark cases out of the public repository.
5. Update source crosswalks only when a chapter packet is actually executed.

Exit criteria:

- source IDs are assigned without orphaning or overloading chapter ownership;
- `docs/book_outline.md` and `book_structure.json` agree;
- source matrix and ownership appendices regenerate cleanly;
- no claim label or support state changes.

### Wave 4 — Decide the durable-semantic-memory boundary

Purpose: determine whether durable semantic memory needs its own chapter after
the existing memory chapters have absorbed their project-derived upgrades.

This wave begins only after Waves 1 and 2. The decision is deliberately late:
improving the Virtual Context ABI, context transactions, claim ledgers,
procedural memory, and artifact graphs may reveal that the remaining unowned
surface is smaller—or more clearly distinct—than it appears now.

The provisional chapter is:

`durable-semantic-memory-and-knowledge-lattices`

Provisional location: Part II, after `virtual-context-abi` and before
`context-transactions-snapshots-mounts-and-taint`.

The chapter may be added only if all of these hold:

- independent primary literature is source-noted for knowledge graphs,
  graph-RAG or graph/vector hybrid retrieval, agent long-term memory, learned
  graph navigation, temporal/event-sourced knowledge, consolidation,
  forgetting, and transactional persistence;
- the chapter owns stable semantic object identity, ontology version/migration,
  provenance-preserving merge/supersession/retraction, graph navigation,
  forgetting/compaction, and restart-consistent persistence;
- the Virtual Context ABI still owns bounded consumer materialization;
- context transactions still own snapshot/mount/taint mechanics;
- procedural memory still owns reusable execution trajectories and tools;
- claim ledgers still own belief support/revision state;
- artifact graphs still own generic artifact/evidence lineage;
- a minimum schema, invalid fixtures, one technical figure, proof/evidence route,
  reader path, and explicit non-claims can ship with the chapter in one coherent
  change.

If these conditions do not hold, reject the new chapter and distribute the
remaining material as sections in the five existing owners. Do not create a
stub to reserve the topic.

Current decision (2026-07-10): **reject a separate chapter for the current
incorporation program**. The ownership test is not met strongly enough to
justify a sixth memory chapter: durable object identity and bounded
materialization remain with the Virtual Context ABI; transactional persistence,
taint, and restart behavior remain with Context Transactions; belief support
and revision remain with Claim Ledgers; reusable execution trajectories remain
with Procedural Memory; and generic lineage remains with Artifact Graphs. The
topic may be reconsidered only if independent literature plus a distinct
interface, invariant family, public-safe artifact, and failure family remain
unowned after those five chapters are re-audited. No placeholder chapter is
added and no support state changes.

### Wave 5 — Reconcile the architecture and roadmap

Purpose: ensure the final book reads as one architecture rather than a series of
historical-project insertions.

Required actions:

- add a single lineage diagram to `integrated-reference-architecture`, showing

- update `prototype-roadmap` so every incorporated concept has a smallest

  and a retirement condition;
- update `open-research-agenda-and-bibliography-plan` with the unresolved
  independent-literature and reproduction gaps;
- remove duplicated definitions introduced during Waves 1–3;
- review part and chapter handoffs for the durable-memory decision;
- preserve all support-state and non-claim boundaries in Human view and future
  reader editions.

## Standard chapter packet

Each chapter is updated as one coherent packet. Do not scatter a concept across
multiple partial edits.

Every packet must include:

1. **Ownership statement:** why this chapter, not an adjacent chapter, owns the
   concept.
2. **Source mapping:** project source IDs plus independent external comparators
   where a generalized claim is made.
3. **Reader-facing prose:** problem, mechanism, non-obvious consequence,
   strongest objection, failure case, and boundary.
4. **Local lineage case:** one concise implementation or contradiction box;
   project names are secondary to the mechanism.
5. **Interface and invariant changes:** exact fields, states, or lifecycle.
6. **Minimum viable artifact:** smallest public-safe schema, trace, fixture,
   proof bridge, or replay that exercises the idea.
7. **Mature endpoint:** how the mechanism composes with the stack without
   implying it already exists.
8. **Negative controls:** at least one attractive invalid case that the design
   must reject.
9. **Evidence disposition:** no-promotion, promotion blocker, demotion, or
   accepted transition reference.
10. **Visual:** reuse or revise the chapter's existing Mermaid diagram where
    possible; add a new figure only when the relationship is genuinely clearer.
11. **Crosswalks:** manifest source IDs, outline queue, Appendix C only if claim
    text/state changes, test/proof plan, implementation horizon, and changelog.
12. **Validation:** proportional validators, Lean when proof code changes, and
    HTML render after a coherent public prose increment.

## Cycle discipline

- Execute one P0 or P1 chapter packet at a time until Wave 1 is complete.
- At most three project-incorporation packets may be open simultaneously: one
  evidence owner, one execution/model owner, and one supporting fixture lane.
- Do not run all project concepts as evidence lanes. Most are argument-level
  source and negative-case improvements.
- Do not create a new planning document for each packet. This roadmap, the
  chapter outline entry, backlog record, and normal evidence/proof surfaces are
  sufficient.
- Do not count a source-ID assignment as incorporation. Incorporation requires
  reader-facing mechanism/failure prose and its owned artifact or exact blocker.
- Do not count a schema as runtime effect, a fixture as external validity, a
  proof field projection as semantic preservation, or a report as capability.
- Finish and validate the current packet before starting another.

## Recommended first six packets

These deliver the most leverage because they prevent later chapter work from
laundering local architecture into capability evidence.

1. **Evidence States:** evidence cell, causal capability chain, honest
   missingness, trainable-state manifest, response-causality receipt.
2. **Benchmark Ratchets:** metric provenance, benchmark closure inheritance,
   representation-capacity gate, retry ceiling.
3. **Artifact Graphs:** canonical projections, transitive revocation,
   replay grades, relocation-safe artifact identities.
4. **Integrated Architecture:** name-to-effect trace and one-lineage diagram.
5. **System Boundaries:** unified authority tuple and execution-domain
   ownership.
6. **Cognitive Compilation:** semantic preservation, ambiguity lineage, repair
   blast radius, and digest-parity negative case.

After these six, reassess the order using actual chapter overlap and validator
cost. Do not promote the durable-memory chapter merely because its source work
is interesting.

## Program acceptance gate

The incorporation program is complete when:

- all P0 and P1 packets have executed or carry an exact rejection/defer reason;
- P2/P3 packets have explicit completed, deferred, or superseded dispositions;
- every used project source ID appears in the owning chapter manifest and
  source crosswalk;
- no repeated project mechanism is counted as independent support;
- all twelve cross-cutting concepts have one primary owner and no competing
  definitions;
- every named transition has a material-effect or exact-blocker statement;
- every model/training claim identifies generated, frozen, trainable, and
  actually updated state;
- every benchmark/metric claim identifies origin, evaluator, target capacity,
  and causal runtime binding;
- upstream quarantine propagates through claims, promotion, readiness, and
  public views in the relevant fixture or remains an explicit blocker;
- the durable-memory chapter has been added as a complete source-grounded
  chapter or explicitly rejected in favor of existing owners;
- `docs/book_outline.md`, `book_structure.json`, Appendix C, Appendix K, source
  appendices, proof manifest, evidence plan, handoffs, and chapter prose agree;
- normal content validation and HTML render pass;
- no support-state promotion is inferred from the incorporation work itself.

## Non-claims

This roadmap does not claim that the historical projects establish model
quality, training effectiveness, external benchmark performance, autonomous
routing, general speech, correct semantic memory, secure execution, hardware
trust, production readiness, safe self-improvement, AGI, or ASI. It does not add
the provisional durable-memory chapter, change a chapter core support state, or
authorize publication of private project material.
