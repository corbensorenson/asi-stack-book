# Cohesive Book Outline

Working title: **The ASI Stack: A Systems Architecture for Governed, Efficient, Self-Improving AI**

Status: architecture outline for drafting the full book. This document defines the intended argument arc, chapter jobs, and bibliography plan. `book_structure.json` remains the ordering source of truth; displayed chapter numbers are generated by Quarto.

## Drafting Thesis

The book argues that advanced AI should not be understood as one larger model, one agent loop, or one benchmark ladder. It should be specified as a governed stack whose layers transform intent into action through explicit boundaries: alignment, governance, planning, memory, reasoning, execution, routing, compression, evidence, and recursive self-improvement.

The stack is efficient because it does not spend maximal cognition everywhere. It routes work, compiles context, reuses artifacts, compresses representations, escalates only when needed, and records residual uncertainty. It is governable because each layer has an authority ceiling, an interface, an invariant set, a failure model, and an evidence record. It is self-improving only when replacement is bounded by stable capability fields, readiness gates, rollback paths, and evaluator integrity.

## Reader Promise

By the end of the book, a technical reader should understand:

- Why ASI should be treated as a systems architecture, not a monolithic model.
- How each layer constrains the others without erasing its separate responsibilities.
- How claims move from design argument to source-derived, prototype-backed, test-backed, or externally supported evidence states.
- How a minimal implementation can grow from artifact graph to governed capability replacement.
- How the living book itself acts as an evidence ratchet instead of a static anthology.

## Book-Level Throughline

1. Define the stack and why model scale alone is not an architecture.
2. Establish efficiency as governed selectivity: routing, compression, memory reuse, and benchmark pressure.
3. Name the failure modes that appear when intelligence is not bounded by authority, evidence, memory discipline, or evaluator integrity.
4. Specify each layer as a boundary with inputs, outputs, invariants, and tests.
5. Integrate the layers into an intent-to-execution reference architecture.
6. Convert the architecture into a staged prototype and living evidence process.

- Explain minimum viable intelligence: use the smallest adequate route that preserves quality.

## Candidate Expansion Slots

The current 16-chapter structure is sufficient for a first full-book drafting goal. These topics should not become new chapters unless drafting pressure proves they need their own boundary:

- Security and adversarial operations: currently distributed across Failure Modes, Governance, Execution OS, and Evidence Ratchets.
- Human interface and delegation: currently distributed across ASI Is a Stack, Alignment, Planning, and Integrated Reference Architecture.
- Deployment operations: currently handled by Prototype Roadmap and Living Book Methodology.
- External literature review: currently handled by Appendix G until it becomes large enough to need its own chapter or part.
- Formal proofs: currently handled by Mathematical and Search Substrates plus Appendices D/E and the `lean/` workspace.

## Drafting Order for the Full-Book Goal

1. Draft Part I first so every later chapter uses the same vocabulary.
2. Draft Part II in interface order: Alignment, Governance, Planning, VCM, Execution, Reasoning, Routing, Compression, Mathematical Substrates, Evidence.
3. Draft Part III after the layer chapters so the integrated architecture can reuse their exact interfaces.
4. Build source notes alongside chapters; do not promote claim support states before source notes exist.
5. Add diagrams and schemas only when they clarify a boundary, invariant, or handoff.
6. Run validation and render after each structural or chapter batch.

## Definition of a Ready Full Draft

A first complete manuscript draft is ready when:

- Every chapter has prose in all standard sections.
- Every assigned source has either a source note or an explicit blocker.
- Every core claim appears in Appendix C with the correct support state.
- No test result is reported unless the test was implemented and run.
- Appendix G reflects the current source corpus and any verified external literature.
- The site renders cleanly from `main`.
