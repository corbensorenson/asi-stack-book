## Thesis

The Circle Calculus AI Contract Suite is a proof-carrying contract layer for finite AI-engineering objects. Its common contract shape maps an engineering object into a finite circular address space, repeated action, stride, period, or window, then attaches theorem ids, Python certificates, schema fields, and explicit non-claims.

## Mechanisms

- Provide guided contract families for RoPE position distinguishability, KV-cache ring-buffer freshness, sparse-attention coverage, looped recurrence schedules, strided candidate fanout, cyclic memory residue/winding, MultiCoil phase features, circulant/block-cyclic mixers, and seed-rule exact regeneration.
- Emit machine-readable receipts, request-validation reports, compact receipt views, bundle checks, artifact manifests, acceptance policies, and downstream rejection reports.
- Use `--require-*` gates so CI can fail on missing status, decision, assurance, pass policy, theorem ids, fields, recommendations, fingerprints, or normalized parameters.
- Require readers to inspect contract status, downstream readiness, evidence fields, optional recommendations, theorem ids, and explicit boundaries/non-claims.
- Treat report labels such as PASS, FAIL, GAPS, and READY as summaries, not substitutes for theorem-linked evidence and claim boundaries.

## Evidence

- The source is a compact usage entry point for Circle's proof-carrying AI contracts.
- It lists schemas, minimum consumer fields, representative report shapes, theorem clusters, CLI commands, generated packs, and reading order.
- It describes structural facts and fixture-level contract checks; it explicitly denies model-quality proof.
- This source note itself did not run Circle contract commands, generated packs, Lean builds, or receipt checks from the ASI Stack repo. A later bounded external receipt slice is recorded separately in `docs/circle_external_receipt_slice.md` for one local rope-position contract replay; that slice does not promote any chapter core claim, downstream consumer claim, or model-quality claim.

## Failure Modes

- Treating READY/PASS labels as model-quality guarantees.
- Ignoring minimum consumer fields, theorem ids, schema fingerprints, or explicit non-claims.
- Treating planner recommendations as quality claims rather than finite contract guidance.
- Accepting non-default model config metadata as covered by a proof frontier when the import/report says it is not.
- Promoting downstream transfer claims before ordinary baselines and named workload reports exist.
