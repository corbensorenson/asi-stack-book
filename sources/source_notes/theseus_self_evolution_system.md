## Thesis

The Theseus self-evolution lane is an evidence-first improvement governor. Its purpose is not random self-mutation; it is to let the local system keep improving while allowing sparse teacher edits only when local evidence reaches a real wall and cheaper interventions have been exhausted.

## Mechanisms

- Apply an intervention ladder: audit benchmark, improve data or bridges, tune training, improve inference/tools/routing, close loops into tools, add adapters or bridge benchmarks, make the smallest architecture change, and add parameters only when evidence requires it.
- Use a self-evolution governor, guarded teacher self-edit runner, ATTD analyzer, benchmark adapter factory, architecture experiment governor, guidance loop, lifecycle governor, deterministic rule/verifier substrates, and outcome ledgers.
- Treat arms, tools, verifier suckers, systems, and some data surfaces as lifecycle-governed cells with renew, observe, improve, split/compress, protect, and retire decisions.
- Use Assembly-Theoretic Technical Debt as a deterministic source-growth governor with GREEN, YELLOW, and RED states.
- Restrict teacher source edits to clean-worktree branch isolation, evidence packets, local checks, and trace ledgers.
- Record repeated successful repairs for future loop closure into local verified tools.

## Evidence

- The source is an implementation-governance note for the Theseus autonomy lane.
- It identifies scripts, reports, configs, dashboards, and machine-readable ledgers that implement or report the lane.
- It contains a concrete governance design for ATTD, teacher edits, benchmark adapter creation, architecture experiments, outcome ledgers, and loop closure.
- No Theseus scripts, ATTD reports, branch-creation flow, teacher runner, or dashboard endpoints were executed from this repo as part of this note.

## Failure Modes

- Teacher edits becoming unbounded architecture invention.
- Autonomous source growth continuing despite repo-health RED states.
- Tool, arm, verifier, or data bloat accumulating without lifecycle review.
- Treating benchmark adapter generation as permission to train on public benchmark answers.
- Confusing outcome ledgers or architecture guidance with actual keep/discard empirical evidence.
